# Rebuild Temper Skill — Design Specification

**Task:** `2026-04-02-rebuild-temper-skill`
**Context:** temper
**Mode:** plan/medium → build (followup task)
**Date:** 2026-04-02

## Problem Statement

The current temper "skill" is actually a command (`~/.claude/commands/temper.md`) — a single
monolithic file generated by `skill.rs` via `format!()`. It documents CLI usage and embeds
workflow logic in one ~190-line blob. This has several problems:

1. **Not a skill** — It doesn't guide agent behavior through enforcement or routing. It's
   reference documentation mixed with workflow instructions.
2. **Context-heavy** — The full file loads into context regardless of which workflow is needed.
   Superpowers already demonstrated that overloading context leads to muddied analysis.
3. **Not extensible** — Adding a new workflow or guidance means editing Rust code, regenerating,
   and reinstalling. No user-side extension point exists.
4. **Domain-blind** — Assumes all work is Rust software development. Subagent guidance uses
   framework-specific examples that don't generalize.
5. **No project fundamentals** — Each session starts cold. The agent discovers constraints
   through friction (user corrections) rather than declared principles.

## Architecture

### File Layout

```
~/.claude/commands/temper.md              # Thin command wrapper (askama-templated)
~/.claude/skills/temper/
├── SKILL.md                              # Router/dispatcher (askama-templated)
├── reference.md                          # CLI command reference (static)
├── subagent-guidance.md                  # 10 universal principles (static)
├── session-lifecycle.md                  # Start/end/drift patterns (static)
├── workflows/
│   ├── build-small.md
│   ├── build-medium.md
│   ├── build-large.md
│   ├── plan-small.md
│   ├── plan-medium.md
│   └── plan-large.md
└── guidance/                             # Extension point (convention-based)
    ├── fundamentals.md                   # Per-context, created by /temper init
    └── (user-added files)
```

### Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| D1 | Command wraps skill | `/temper` is the natural entrypoint users reach for. The command parses args and invokes the skill. All intelligence lives in the skill. |
| D2 | Modular with convention-based discovery | Agent reads only what it needs. New files in `workflows/` or `guidance/` are picked up by convention — no registry, no code changes. |
| D3 | Hybrid templating | Askama for files with dynamic content (SKILL.md, command wrapper). `include_str!()` for static content (everything else). Matches existing template patterns in `templates.rs`. |
| D4 | Universal subagent guidance | 10 principles written domain-agnostically with wrong→right tables. No framework-specific examples in the core guidance. |
| D5 | Domain-aware routing | Skill infers or asks what kind of work before dispatching. Applies guidance subset accordingly — software gets all 10, non-software gets the universal subset. |
| D6 | Plugin-aware, not plugin-dependent | Workflows reference external skills (brainstorming, TDD, writing-plans) conditionally. Everything works standalone. |
| D7 | Memory-aware plugin discovery | Checks auto-memory for user preferences, scans installed skills, asks before applying. |
| D8 | Project fundamentals via guided init | `/temper init` creates `guidance/fundamentals.md` through interactive flow with opinionated but customizable defaults. |
| D9 | `--path` override | `temper skill install --path <dir>` for non-default install locations. |
| D10 | Session lifecycle as first-class | Start/end patterns, drift detection, checkpoint prompts — all in a dedicated file. |

## Component Specifications

### 1. SKILL.md — The Router

The only file loaded initially. Responsibilities:

- Declare trigger conditions via frontmatter description
- Route by mode/effort to the correct workflow file
- Check auto-memory for user plugin preferences
- Discover installed skills and ask which to apply
- Point to supporting files without force-loading them

**Frontmatter:**

```yaml
---
name: temper
description: Use when managing knowledge vault tasks, sessions, or search —
  task start/create/done, session save, semantic search, context discovery,
  or any /temper command invocation
---
```

**Templated fields:** config hash, vault path, context list.

**Routing logic:**

1. Parse the invocation (task start, session save, search, etc.)
2. For task start:
   a. Read task content → extract mode and effort
   b. If mode/effort missing, ask briefly
   c. Infer or ask domain: software, writing, research, design, other
   d. Check `guidance/fundamentals.md` — if missing, offer `/temper init`
   e. Check auto-memory for plugin preferences
   f. Scan installed skills, ask user which to apply
   g. Load `workflows/{mode}-{effort}.md` and follow it
3. For other commands: delegate to reference.md patterns

### 2. Subagent Guidance (`subagent-guidance.md`)

Ten universal principles in two tiers.

**Foundational (SG-1 through SG-6):**

- **SG-1: Follow Existing Patterns** — Before writing anything, read the file you're
  modifying AND a sibling in the same module. Match the style you find: naming, imports,
  structure, error handling. Don't invent new patterns.
- **SG-2: Single Responsibility** — Each function does one thing. If it constructs AND
  processes AND formats — split it. Follow the project's existing layering.
- **SG-3: No Logic Duplication** — Would two implementations drift independently over time?
  Extract. Otherwise leave inline. Don't create premature abstractions for one-time operations.
- **SG-4: Test Strategy** — Unit tests co-located with code. Integration tests separate.
  One behavior per test with descriptive names. Tests must actually run — verify, don't assume.
- **SG-5: Don't Over-Build** — Implement exactly what the task says. No speculative features,
  no defensive code for impossible cases, no "nice to have" extras.
- **SG-6: Verify Before Claiming Done** — Run the verification command. Read the output.
  Don't claim success based on what you think the code does.

**Friction-Derived (SG-7 through SG-10):**

- **SG-7: Prefer Native Solutions** — Don't invent when the framework, language, or platform
  provides. If a proper tool exists, use it over a hand-rolled alternative.
- **SG-8: Front-Load Constraints** — Before proposing anything: (1) existing abstractions?
  (2) platform/deployment limits? (3) async/performance requirements? List findings before writing.
- **SG-9: Don't Dismiss Owned Failures** — If the user owns both sides of an interaction,
  debug the full stack. Never declare "not our problem" without proving external causation.
- **SG-10: Checkpoint Before Continuing** — After each major step, report: what's done,
  what's next, any concerns about approach drift.

**Each principle includes a wrong→right table** using domain-neutral language:

```
| Wrong                                          | Right                                        |
|------------------------------------------------|----------------------------------------------|
| Silently swallow errors with defaults          | Return specific errors with context           |
| Build a new abstraction for one use            | Inline it, extract later if repeated          |
| Claim "tests pass" without running them        | Run, read output, report result               |
| Propose complex solution without checking      | List existing tools/abstractions first         |
| Declare a failure "not our problem"            | Prove external causation before dismissing     |
```

**Domain-aware application:**

- Software tasks: all 10 principles apply
- Non-software tasks: SG-1 (follow patterns), SG-5 (don't over-build), SG-6 (verify),
  SG-10 (checkpoint) apply. The rest are software-specific.

**Dispatch instruction:** "When dispatching subagents, include all applicable principles
verbatim in the subagent prompt. Do not summarize or paraphrase."

### 3. Session Lifecycle (`session-lifecycle.md`)

**Session Start:**

1. Check recent sessions: `temper session list --context <current>`
2. If resuming work, read the last session note for continuity
3. Search for relevant context: `temper search "<topic>"`
4. If `task start`, load task and route by mode/effort

**Session End:**

All workflows converge on session save via stdin:

```bash
cat <<'EOF' | temper session save "<title>" --task <slug> --state <state>
## Goal
What we set out to do

## What Happened
Key actions, decisions, and outcomes

## Decisions
Choices made and why

## Connections
Related tasks, concepts, or contexts touched

## Next Steps
What to pick up next session
EOF
```

Emphasis: always pipe content via stdin. Without stdin, the command creates placeholder
boilerplate that's useless as a session record.

**Mid-Session Drift Detection:**

| Signal | Likely Drift | Action |
|--------|-------------|--------|
| build/small needing design decisions, touching 3+ areas, multiple approaches | Effort too low | Suggest build/medium |
| build/medium needing decomposition into multiple deliverables | Effort too low | Suggest build/large, create sub-tasks |
| plan/large with obvious first task, roadmap has 1-2 items | Effort too high | Suggest plan/medium or start building |
| Software task hitting non-software questions | Domain mismatch | Pause, reassess scope |

On confirmation: `temper task move <slug> --mode <new> --effort <new>`

**Checkpoint Pattern:**

For medium and large efforts, prompt after each major step:

> "Checkpoint: (1) What's done, (2) What's next, (3) Any concerns about approach drift,
> (4) Does anything conflict with project fundamentals?"

### 4. Workflow Files (`workflows/`)

Each file is a self-contained playbook. Common shape:

```markdown
# {Mode}/{Effort} Workflow

## When This Applies
Brief description.

## Steps
Numbered steps with clear actions.

## Completion
Session save pattern, task state transition.
```

**Workflow specifics:**

| File | Key Behavior |
|------|-------------|
| `build-small.md` | No brainstorming. Implement → test → commit → session save. Fastest path. |
| `build-medium.md` | Discovery → brainstorm → plan → implement. References external skills conditionally. Inline alternative if skills unavailable. |
| `build-large.md` | Like medium but multi-session. Creates sub-tasks. Each session: work one task, save session, create next. |
| `plan-small.md` | Quick research → write findings → session save. No brainstorming. |
| `plan-medium.md` | Discovery → brainstorm problem space → design spec → session save. |
| `plan-large.md` | Deep discovery → brainstorm to map problem space (NOT design implementation) → goal roadmap → first actionable task. Code only if user pushes. |

**Plugin-aware, not plugin-dependent:**

Each workflow that would invoke an external skill uses conditional language:

> "If the user has opted into [skill], invoke it now. Otherwise, follow the inline
> alternative below."

The plugin discovery step in SKILL.md has already resolved which skills are available
and the user has confirmed preferences.

**Extension:** Custom workflows follow `{mode}-{effort}.md` naming in `workflows/`.
SKILL.md routes by filename match — if the file exists, it gets used.

### 5. Project Init & Fundamentals

**Trigger:** `/temper init` or auto-detected on first `task start` in a context without
`guidance/fundamentals.md`.

**Guided flow (interactive, one question at a time):**

**Domain & Stack**
- What kind of work happens in this context? (software, writing, research, mixed)
- If software: primary language(s), framework(s), deployment target(s)
- If non-software: what tools/formats are central?

**Principles**
- Offer opinionated defaults appropriate to the domain:
  - Software: composition over inheritance, fail loudly, SRP, prefer native solutions,
    verify before done
  - Writing: clear structure before prose, one idea per paragraph, verify claims against
    sources, don't pad
  - Research: state hypothesis before investigating, cite sources, distinguish observation
    from inference
- Ask: "These are the defaults. Adopt all, pick some, or write your own?"

**Quality Gates**
- What verification looks like (test, lint, build commands for software; review checklist
  for non-software)
- Whether code review skills should auto-invoke
- Whether TDD or other discipline skills are expected

**Boundaries**
- Off-limits: files, patterns, approaches to avoid
- Platform/deployment constraints

**Output:** `guidance/fundamentals.md` with frontmatter (context, domain, stack) and
sections for principles, quality gates, and boundaries.

### 6. Reference File (`reference.md`)

Clean, scannable CLI reference. No workflow logic:

- Invocation rules (always `temper` from PATH)
- Command table with syntax and descriptions
- Stages, mode/effort definitions
- Discovery workflow: search → context → targeted reads
- Template access via `--show-template`

Extracted from the top half of the current command file.

### 7. Command Wrapper (`~/.claude/commands/temper.md`)

Minimal — frontmatter + skill invocation:

```markdown
<!-- config-hash: {hash} -->
---
name: temper
description: Knowledge vault operations — invoke the temper skill
---

Invoke the temper skill to handle this request.

ARGUMENTS: $ARGUMENTS
```

## Rust Implementation Scope

### Files to modify

- `crates/temper-cli/src/commands/skill.rs` — `generate()` becomes
  `generate_skill_directory()`. `install()` writes full directory tree + command wrapper.
  `check()` validates directory structure + hash staleness. Add `--path` argument.
- `crates/temper-cli/src/main.rs` — Wire `--path` argument to install subcommand.
- `crates/temper-cli/src/templates.rs` — Add `SkillTemplate` and `CommandTemplate` structs.
- `crates/temper-cli/Cargo.toml` — No new dependencies expected.

### Files to create

- `crates/temper-cli/templates/skill.md` — Askama template for SKILL.md
- `crates/temper-cli/templates/command.md` — Askama template for command wrapper
- `crates/temper-cli/skill-content/` — Directory of static `.md` files:
  - `reference.md`
  - `subagent-guidance.md`
  - `session-lifecycle.md`
  - `workflows/build-small.md`
  - `workflows/build-medium.md`
  - `workflows/build-large.md`
  - `workflows/plan-small.md`
  - `workflows/plan-medium.md`
  - `workflows/plan-large.md`

### Templating strategy

| File | Method | Interpolated fields |
|------|--------|-------------------|
| SKILL.md | Askama template | config hash, vault path, context list |
| commands/temper.md | Askama template | config hash |
| Everything else | `include_str!()` | None — static content |

### Install behavior

`temper skill install [--path <dir>]`:

1. Resolve output directory (default `~/.claude/skills/temper/`, override via `--path`)
2. Create directory structure: `workflows/`, `guidance/`
3. Render and write SKILL.md (askama)
4. Write static files from `include_str!()` — these are **always overwritten** on install
   (they are managed content, not user content)
5. Render and write command wrapper to `~/.claude/commands/temper.md`
6. Do NOT overwrite files in `guidance/` (user-created content preserved)
7. The `guidance/` directory is created if missing but never cleaned — install is additive

`temper skill check`:

1. Verify skill directory exists with expected structure
2. Check config hash staleness in SKILL.md
3. Check command wrapper exists and hash matches
4. Report missing files

## Future Considerations

- **tasker-mcp integration:** This skill architecture will be the foundation for the
  temper-mcp crate. When MCP tools are built, the skill and CLI remain the preferred
  interface for flexibility. MCP extends capabilities but does not replace the skill/CLI
  workflow. A future task should fold MCP awareness into the skill's plugin discovery step.
- **Cloud sync of skill files:** Not in scope. Skill files are local to the machine.
  `guidance/fundamentals.md` could eventually sync through the vault, but that's a
  separate concern.

## Out of Scope

- Custom workflow creation UI (document the convention, users create files manually)
- Auto-migration from current command format (manual `temper skill install` is sufficient)
- MCP server implementation (future task, this skill will be its foundation)
