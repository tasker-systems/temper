# temper-ts: into the drift regime

**Date:** 2026-07-13
**Goal:** `019f4910-b2b3-7fc0-804d-b008fd909bee` — "temper-rb — a native Ruby client for the temper API", whose stated scope is the SDK *family* (`temper-py`, `temper-ts`), which "inherit the contract work for free". This is that inheritance, collected.
**Status:** approved

## The problem

`temper-ts` is the one first-party client that **cannot be broken by a contract change**.

`temper-rb` is a full SDK: a 181-file generated core under `Temper::Generated::` emitted from
`openapi.json` by a pinned generator, a ~955-line hand-written skin, a drift gate wired into
`cargo make check` (`openapi-rb-drift`), and a never-skipping CI backstop (`test-ruby` → `rake drift`).
A DTO change in Rust that leaves the gem stale turns something red.

`temper-ts` is not an SDK. It is an **auth layer**: `credentials.ts` (the `ClientCredentials` M2M
mint), the mock issuer/API for tests, and a `TEMPER_TS_VERSION` constant. Its only consumer — the
steward — reaches temper over MCP, not over the HTTP API, and imports exactly `BearerToken`,
`ClientCredentials`, `Credentials`, `TokenResult`, `TEMPER_TS_VERSION`, and `temper-ts/testing`. Its
CI job (`test-agents-ts`) runs typecheck and vitest and has **no relationship to `openapi.json`
whatsoever**.

That is fine only while nothing calls the API from TypeScript. Callers are coming. This lands
temper-ts inside the drift regime *before* it accumulates hand-written surface that would then have
to be retrofitted.

## The finding that shaped the design

The premise we started from — "ts-rs generates some types, OpenAPI generates the others, together
they give the full picture" — is **false**, and measurably so.

Of the 133 types ts-rs exports into `packages/temper-ui/src/lib/types/generated/`, **103 also exist
as OpenAPI schemas**: `ManagedMeta`, `ResourceRow`, `TeamRole`, `SearchResultRow`, `Profile`,
`AtlasNode`, and ninety-seven more. The two generators do not divide the world; they overlap it by
77%. The genuinely-disjoint parts are real but small — 30 types only ts-rs emits (the non-HTTP wire
types: `ClaimedEmbedJob`, the `Relationship*` events, `DocType`, `RuntimeBinding`) and 58 schemas
only OpenAPI has (request/response DTOs the UI never needed).

So a `schema.ts` in temper-ts means two first-party TypeScript clients of one API, typed by two
generators with different conventions. That is the bug class this arc keeps fighting.

**It does not sink the design; it sharpens it.** ts-rs emits *types*. It does not emit *operations* —
it has no paths, verbs, params, or status codes, and structurally cannot. "The rest of the client is
just HTTP verbs" is only true if something tells you what the verbs are, and `openapi.json`'s `paths`
object is the only artifact in this repo that does. The duplicated **type** half is the price of the
**operations** half, and the duplication is bounded: both generators derive from the same Rust structs
under gates, so neither drifts from Rust silently. What they can do is drift from each other in
*spelling*, which bites the day temper-ui and temper-ts want to share code.

Hence the exit, stated rather than tolerated: **temper-ui will import its HTTP-contract types from
temper-ts's `schema.ts` and retire the 103 overlapping ts-rs types.** It does not do so today only
because SvelteKit does not consume the API as a standard fetch-style client, and until the API was
stable, ts-rs was how type consistency was held. That migration is out of scope here — but because it
is the direction, `schema.ts` is a **public export of temper-ts from day one**, not an internal
detail. An export the future consumer cannot reach is not a direction, only an intention.

## Decisions

### D1 — `openapi-typescript`, not `openapi-generator`

The gem is generated by `openapi-generator` (pinned 7.23.0, Docker or Java). The obvious symmetric
choice is its `typescript-fetch` target: one generator pin, one script shape, one mental model, and
`temper-py` later takes the same road.

**Rejected.** The gem uses a *generated HTTP layer* because Ruby has no native one worth having.
TypeScript does — `fetch` is in the runtime — so the generated **client** buys almost nothing while
the generated **types** are the entire point. `typescript-fetch` would emit ~170 files of
`FromJSON`/`ToJSON` boilerplate into a package that has five source files today, and would risk
rotting into a second, worse HTTP client sitting next to `credentials.ts`. Its OpenAPI 3.1 handling
is also its weakest area, and our spec is 3.1.

`openapi-typescript` is 3.1-native and emits **one types-only file**. Run against the real
`openapi.json` (68 paths, 161 schemas, 325KB) it produced a 7,402-line `schema.ts` in **71ms**, with
no Docker and no Java: it carries the Rust doc comments through as JSDoc, renders the spec's
`oneOf[null, $ref]` nullable style as clean `field?: T | null`, and exports three top-level maps —
`paths`, `components`, and `operations`.

The cost is a **tooling delta**: a second generator to pin, and no generated API-class layer. Both are
accepted. Parity here is of *regime* (a contract-derived artifact under a drift gate), not of *shape*.

### D2 — the artifact is public

`clients/temper-ts/src/generated/schema.ts`, committed. Emitted as `.ts` and not `.d.ts` so `tsc`
carries it into `dist/` on the normal build. `package.json`'s `exports` gains `"./schema"`, and
`index.ts` re-exports `paths` / `components` / `operations`. See the exit above.

### D3 — the pin lives in the lockfile, the invocation lives in the script

`openapi-typescript` is pinned to an exact version (no caret) in temper-ts's `devDependencies` and
locked in `package-lock.json` — TypeScript's native pinning mechanism, rather than transplanting the
gem script's `GENERATOR_VERSION` constant. `.github/scripts/generate-temper-ts.sh` mirrors its Ruby
sibling as the single source of truth for the *invocation* (input, output, flags), called from all
three places that must agree: `cargo make openapi`, the drift check, and CI. It runs `npm ci` so the
pinned version — not a resolved one — is what generates.

It needs Node and nothing else. The gem's script is deliberately toolchain-light so a Rust developer
who changed a DTO can regenerate without standing up Ruby; the TypeScript equivalent of that virtue is
cheaper still.

### D4 — a minimal client, with no per-endpoint methods

The gem hand-writes `Resources`, `Contexts`, `CognitiveMaps` because Ruby has no type inference —
someone must write `create(title:, context:)` by hand. TypeScript has inference:
`openapi-fetch`'s `createClient<paths>` makes all 68 paths callable with typed params and **per-status
typed responses**, straight off `schema.ts`. Hand-writing `resources.create()` over that would add a
second, worse spelling of something already correct.

**Generate what can be generated; hand-write only what cannot.** What cannot is auth and attribution:

- **`src/auth-fetch.ts`** — `createAuthedFetch({ credentials, fetch? })` returns a `fetch`-shaped
  function that sets `Authorization: Bearer <token>` and `X-Temper-Surface: sdk`, and re-mints
  **once** on a 401 when `credentials.canRefresh`. This is the steward's `temperFetch`
  (`agent/lib/temper-auth.ts`) lifted out of the steward, minus what is genuinely eve's: the env
  names, the Vercel Connect strategy, and the 5xx cold-start retry. It takes an **inner `fetch`** so a
  caller can keep its own retry underneath the auth layer.

  **The surface is hardcoded `sdk`, with no override.** The server trusts `{sdk, cli}` and attributes
  a `cli` write to the `<handle>@cli` emitter — so an override would be a knob for writing a lie into
  the event ledger, and there is no TypeScript CLI to tell the truth with. The gem hardcodes it too.

  **The steward's migration onto this is a follow-up, not a rename.** Its `fetchWithRetry` is
  `(url, init)`-shaped while `FetchLike` is `(Request)`-shaped, so composing them needs the steward's
  retry reshaped first — `createAuthedFetch({ credentials, fetch: fetchWithRetry })` does not compile
  as written. Until that lands the steward keeps its own hand-rolled 401 re-mint (which carries the
  same over-minting bug fixed here, and sends no surface header at all).

- **`src/client.ts`** — `createTemperClient({ baseUrl, credentials, fetch? })`, handing that fetch to
  `createClient<paths>`.

`canRefresh` — the `Credentials` member the publish task flags as "must settle before first publish" —
is precisely what the 401 branch consumes. Building this client is what *validates* it.

**No error hierarchy — but the client is not throw-free, and the difference matters.** `openapi-fetch`
returns `{ data, error, response }` with `error` typed per-status **from the spec**, so HTTP status
errors need no hierarchy of ours; the gem needs `errors.rb` only because Faraday raises. What does NOT
arrive in `error` is anything the *auth layer* throws: `openapi-fetch` registers no `onError`
middleware, so a `TokenMintError` (a wrong client secret — the single most common M2M
misconfiguration) or a network `TypeError` propagates out of the call. Callers need a `try` as well as
an `if (error)`, and the docs say so. Inventing a hierarchy would be inventing drift; pretending
nothing throws would be inventing a lie.

**One new runtime dependency:** `openapi-fetch` (~6kb, zero deps of its own). temper-ts has zero
runtime deps today and the steward bundles it into a serverless function, so this is stated rather
than slid in.

### D5 — the gates, and the CI trigger set that would have silently defeated them

- `cargo make openapi` regenerates spec → gem → schema. Unlike the gem step (which **skips** without
  Docker), the TypeScript step never skips, so `cargo make check` gains a drift gate that is a real
  gate rather than a best-effort one.
- New `openapi-ts` and `openapi-ts-drift` cargo-make tasks; `openapi-ts-drift` joins the `check` chain
  beside `openapi-rb-drift`.
- `test-agents-ts.yml`'s temper-ts job gains a drift step — the never-skipping backstop, the role
  `test-ruby` plays for the gem.
- **`detect-ci-scope.sh` must add `^openapi\.json$` to `test-agents-ts`'s trigger set.** Today it is
  `^clients/temper-ts/|^packages/agent-workflows/|^tests/contracts/|^\.github/workflows/test-agents-ts\.yml$`
  — no `openapi.json`, which is correct while temper-ts is inert to the contract and **wrong the
  instant it has a drift gate**: a Rust DTO change would regenerate the spec and never run the job
  whose gate catches the stale schema. `test-ruby`'s set already carries `^openapi\.json$` with a
  comment saying it is there "precisely because a contract change must" run the gem's gate. This is
  the same rot the previous session found in `tests/contracts/` not being in `test-ruby`'s trigger
  set — a gate that runs nowhere.

  `crates/**` deliberately stays **out** of the set: the spec is committed, and `openapi-check` in
  `code-quality` already forces a Rust DTO change to land a regenerated `openapi.json` in the same PR.
  `openapi.json` in the trigger set is therefore both sufficient and precise.

## Verification

Evidence, not assertion:

1. **The scope fix is proven by a failing test first.** Add a case to
   `.github/scripts/test-detect-ci-scope.sh` asserting an `openapi.json`-only change sets
   `run-test-agents-ts=true`. Watch it fail against the current script, then fix the regex.
2. **The drift gate bites.** Add a field to `openapi.json`, run `cargo make openapi-ts-drift`, confirm
   it goes red, restore `openapi.json` **from a file copy** (never `git checkout` — it would also
   revert the work in progress).
3. **The 401 re-mint is exercised end to end.** A vitest against the existing `startMockApi` /
   `startMockIssuer` harness: a 401 followed by a 200 mints exactly once and retries once; a
   `BearerToken` (cannot refresh) gets its original 401 back **untouched**, body and all.
4. `schema.ts` typechecks under temper-ts's strict tsconfig and lands in `dist/`.
5. The existing 29 temper-ts tests and 15 steward tests stay green.

## Out of scope

### Rejected

- **`openapi-generator typescript-fetch`** — see D1. Not "later"; not the right tool for this language.
- **A hand-written error hierarchy** — see D4. The contract's typed per-status errors are the model.
- **Per-endpoint domain modules** (`resources.create()`, `contexts.list()`) — see D4. They would
  duplicate what inference already gives, and any shape chosen now would be a guess.

### Deferred

- **Publishing to npm** — its own task (`019f5e00`), staying in backlog. It carries `canRefresh`'s
  settlement, `TEMPER_TS_VERSION`'s reconciliation, retiring the steward's `file:` bridge, and turning
  the Vercel outside-root toggle back off. None of it blocks the drift regime, and the drift regime
  does not need it.
- **temper-ui's migration off the 103 overlapping ts-rs types** — the named exit above. It wants the
  `file:`-vs-published question answered first (temper-ui is a bun workspace member; temper-ts
  deliberately is not), which is the publish task's business.
- **`refs.ts` (`parse_ref`), the `Act` authorship/correlation sugar, and the domain modules** — shapes
  that the incoming callers should name, not shapes we should guess.
- **The steward's migration onto `createAuthedFetch`** — surfaced by review, deliberately not bundled.
  It is not a rename (its `fetchWithRetry` must be reshaped to `FetchLike` first), and it changes a
  *deployed* agent's provenance: today the steward's REST writes carry **no** `X-Temper-Surface` and
  therefore attribute to `web`; composing this client would move them to `sdk`. That is a real change
  to the event ledger's authorship history and wants a deliberate decision, not a drive-by. It also
  carries the same over-minting bug fixed here (its hand-rolled 401 handler calls `refresh()`
  unconditionally), which is the substantive reason to do it soon.
- **Hardening `check-temper-rb-drift.sh`** — it has the same weakness the TS gate was hardened against
  (`git diff --exit-code` exits 0 over a path matching nothing), and is *more* exposed: it diffs a
  whole directory, so a generator config change that relocated the output would silently empty the
  gate rather than fail it. Left alone here to keep this PR's blast radius on the TS side.
