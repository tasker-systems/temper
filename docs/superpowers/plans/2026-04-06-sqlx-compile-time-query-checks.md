# Migrate sqlx::query() to sqlx::query!() Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Migrate all runtime `sqlx::query()` calls to compile-time checked `sqlx::query!()` / `sqlx::query_as!()` / `sqlx::query_scalar!()` macros so that SQL errors are caught at compile time rather than at runtime.

**Architecture:** Mechanical migration of ~82 queries across 9 service files, 1 middleware file, 2 test fixture files, and 1 e2e test file. Custom SQL functions (`can_modify_resource`, `persist_resource_chunks`, `replace_resource_chunks`, `sync_diff_for_device`, `contexts_visible_to`, `resources_visible_to`, `unified_search`) need type annotations in the macros. JSONB columns (`payload`, `managed_meta`, `open_meta`, `preferences`, `vault_config`) need explicit `as "field: Json<Value>"` casts. After migration, generate `.sqlx/` cache for offline compilation in CI.

**Tech Stack:** Rust, sqlx 0.8 macros, PostgreSQL 18 + pgvector, cargo-sqlx CLI

**Key decisions:**
- `query_as!()` replaces `query_as::<_, T>()` — the macro generates an anonymous struct, so we use field-by-field mapping via `query!()` + manual construction instead, OR we keep `query_as!()` which requires the struct fields to match column names exactly.
- Tuple returns like `(Uuid,)` and `(String,)` become `query_scalar!()` or `query!()` with named field access.
- Test files (fixtures, e2e) get the same treatment — SQL correctness matters everywhere.
- `.sqlx/` cache committed to git for CI offline compilation.
- CLAUDE.md updated: `SQLX_OFFLINE` allowed for compilation; `DATABASE_URL` required for test execution.

## Preliminary: Understanding the macro differences

| Before | After | Notes |
|--------|-------|-------|
| `sqlx::query("SQL").bind(x).execute(pool)` | `sqlx::query!("SQL", x).execute(pool)` | Binds become macro args |
| `sqlx::query_as::<_, T>("SQL").bind(x)` | `sqlx::query_as!(T, "SQL", x)` | Type must match column names exactly |
| `sqlx::query_scalar("SQL").bind(x)` | `sqlx::query_scalar!("SQL", x)` | Returns single scalar |
| `sqlx::query_as("SQL").bind(x)` (tuple) | `sqlx::query_scalar!("SQL", x)` or `sqlx::query!("SQL", x)` | Tuples become named fields |
| JSONB bind: `.bind(&payload)` | Macro arg: `payload as _` | sqlx needs type override for JSONB |
| `$3::vector` cast in SQL | `$3 as _` in macro | Custom type casts use `as _` |

**Important:** `sqlx::query_as!(T, ...)` requires that struct `T` has fields matching the query's column names exactly (no aliasing mismatch). All our `FromRow` structs already match because they were designed for `query_as::<_, T>()` with explicit column lists. The macros will validate this at compile time.

**JSONB handling:** For `serde_json::Value` parameters bound to JSONB columns, the macro needs a type override. The pattern is: `$N as _` is not needed — sqlx 0.8 infers `Json<Value>` from the Rust type when using `&serde_json::Value`. However, for **nullable** JSONB or custom types, we may need `field as "field: _"` in the SELECT list.
